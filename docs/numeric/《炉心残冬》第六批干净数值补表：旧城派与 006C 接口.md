## 一、文件使用规则  
本文件用于整理旧城派系统与 006C 誓言 / 铁腕炉约接口相关数值，供代码窗先接入可运行骨架，并供测试窗进行压力测试与平衡复核。  
本文件不回改任何已封存 Patch 正文。  
若本文件与以下文件冲突，按以下优先级处理：  
```
全局修正总控
>
A 干净封存版
>
无 A 版时对应的干净封存版
>
本批干净数值补表
>
小修、覆盖规则、返工清单、旧补表

```
本文件中数值分为：  
```
已封存 / 可直接实现：
来自全局总控、A 版封存正文或用户后续明确确认口径，当前无冲突，可交代码窗实现。

代码窗暂定实现值 / 测试窗必测：
允许代码窗先写入配置表或常量表，用于跑通游戏骨架。
不等于最终平衡封存。
必须由测试窗试玩、压力测试后确认保留、调整或封存。

后置 / 待确认 / 不进代码窗：
当前不写入代码实现，不由代码窗自行猜测。

```

## 三、本批已修正的问题  
**3.1 守炉堂 / 巡查所口径按全局总控覆盖**  
旧第六批曾写：  
```
守炉堂 / 巡查所占普通区域槽位。
守炉堂 / 巡查所需要玩家手动 build。
守炉堂 / 巡查所不会随路线入口签署自动生成。
守炉堂 / 巡查所成本 45 木材，20 钢铁。

```
本批作废该旧口径。  
按全局总控统一为：  
```
守炉堂 / 巡查所为 006C 特殊核心建筑。
不占普通建筑槽位。
不需要玩家手动 build。
路线入口签署后自动生效。
必须显示在面板上。
至少 1 名可工作人口值守即可运行。
不可 heat。
不参与普通建筑低温停工。

```
说明：  
```
守炉堂 / 巡查所仍然是路线核心承接物。
但 V1 不把它们作为普通建筑建造链处理。
代码窗不得实现守炉堂 / 巡查所建造成本、占槽、build 队列或建筑选址。

```
   
⸻  
   
**3.2 终火誓约 / 最高戒令失败轴边界压回全局口径**  
本批统一：  
```
终火誓约只承接信任失败轴。
最高戒令只压制恐慌失败轴。
二者不删除信任 / 恐慌数值。
二者不免疫人口归零。
二者不免疫炉心崩毁。
二者不免疫饥饿、寒冷、疾病、死亡。
二者不新增或删除第七霜落硬失败。

```
硬失败仍只保留：  
```
存活人口 = 0。
炉心崩毁。
信任 = 0 且未被终火誓约承接。
恐慌 = 100 且未被最高戒令压制。

```
作废：  
```
第七霜落未撑过仍然失败。
终火誓约让第七霜落低分不失败。
最高戒令让第七霜落低分不失败。
终火誓约 / 最高戒令免疫资源崩溃。
终火誓约 / 最高戒令免疫人口死亡。

```

## 五、第 24 天：南方来信与初始化  
**5.1 触发**  
已封存 / 可直接实现：  
```
第 24 天固定触发“南方来信”。
触发后开启旧城派系统。
该事件不是普通随机事件。
该事件每局只触发 1 次。

```
代码窗字段：  
```
old_city_faction_trigger_day = 24
old_city_faction_trigger_event = "southern_letter"
old_city_faction_active = true after event resolved

```
**5.2 参考人口**  
代码窗暂定实现值 / 测试窗必测：  
```
old_city_faction_reference_population = current_alive_population_at_day24

```
说明：  
```
旧城派阈值以第 24 天触发时的存活人口作为参考人口。
后续人口增减不会反复移动阶段阈值。
这样玩家能理解旧城派目标，而不是每天阈值漂移。

```
**5.3 初始人数公式**  
代码窗暂定实现值 / 测试窗必测：  
```
old_city_initial_base = max(8, ceil(old_city_faction_reference_population * 0.08))

```
第 24 天选项：  
```
A：公布来信
old_city_faction_count = old_city_initial_base + 2
trust_change = +1
panic_change = +2

B：压下来信
old_city_faction_count = old_city_initial_base
trust_change = -2
panic_change = +1
old_city_hidden_growth_bonus_days = 3
old_city_hidden_growth_bonus_per_day = +1

```
说明：  
```
公布来信提高透明度，但扩大传言传播。
压下来信短期人数较低，但形成地下增长。
两者都不直接导致失败。

```

## 六、旧城派阶段阈值  
代码窗暂定实现值 / 测试窗必测：  
```
old_city_low_threshold = max(10, ceil(reference_population * 0.12))
old_city_mid_threshold = max(18, ceil(reference_population * 0.22))
old_city_high_threshold = max(28, ceil(reference_population * 0.35))

```
示例：  
```
若第 24 天存活人口为 100：
低阶阈值 = 12
中阶阈值 = 22
高阶阈值 = 35

若第 24 天存活人口为 80：
低阶阈值 = 10
中阶阈值 = 18
高阶阈值 = 28

```
阶段推进：  
```
阶段 1：第 24 天南方来信，初始化旧城派人数。
阶段 2：old_city_faction_count >= low_threshold，暗中传言。
阶段 3：old_city_faction_count >= mid_threshold，公开集结。
阶段 4：old_city_faction_count >= high_threshold，离城倒计时。
阶段 5：倒计时结束、旧城派被压低、或第七霜落前最终结算窗口。

```
限制：  
```
每个阶段事件每局最多触发 1 次。
阶段事件不会绕过旧城派人数池直接判负。
阶段 4 不等于立即出走。
最终出走仍由倒计时或最终结算触发。
第 49 天后不再开启新的旧城派阶段。
第七霜落期间不再开启新的离城倒计时。

```
   
⸻  
   
## 七、旧城派每日增长 / 下降规则  
**7.1 基础增长**  
代码窗暂定实现值 / 测试窗必测：  
```
old_city_base_daily_growth = +1
old_city_daily_growth_cap = +6
old_city_daily_decline_cap = -3

```
规则：  
```
旧城派激活且未 resolved 时，每日 end_day 后计算趋势。
每日增长 / 下降不会在第 24 天触发当天重复结算。

```
**7.2 压力增长来源**  
每日增长修正：  
```
trust < 40：+1
trust < 25：额外 +1

panic > 65：+1
panic > 80：额外 +1

food_days < 1.5：+1
homeless_population >= 20：+1
medical_gap >= 5：+1
unhandled_bodies >= 6：+1
deaths_today >= 3：+1

ordinary_promise_failure_today：+2
serious_promise_failure_today：+4
critical_promise_failure_today：+6

```
每日增长上限：  
```
每日由长期状态造成的增长，最多 +6。
固定增员事件、旧城派阶段事件选项、006C 主动行动造成的变化，不计入每日增长上限。

```
**7.3 稳定下降来源**  
若同时满足：  
```
trust >= 60
panic <= 40
food_days >= 2.0
homeless_population = 0
medical_gap = 0
unhandled_bodies = 0

```
则：  
```
old_city_faction_count_change = -1

```
若同时满足以上条件，且近 3 天无承诺失败、无重大死亡事件、无拒绝固定增员：  
```
old_city_faction_count_change = -2

```
说明：  
```
城市长期稳定可以自然压低旧城派，但速度慢。
玩家不能只靠自然下降快速清空旧城派。

```
**7.4 固定增员与旧城派**  
沿用第五批固定增员接口，并统一：  
```
旧城派未激活前，固定增员不直接增加旧城派人数。
旧城派已激活后，固定增员可影响旧城派人数。

```
第 37 天后期难民潮旧城派接口：  
```
全部接纳：
old_city_faction_change = -5

部分接纳：
old_city_faction_change = +4

拒绝接纳：
old_city_faction_change = +10

```
状态：  
```
代码窗暂定实现值 / 测试窗必测。

```

```
old_city_faction_countdown_day = min(current_day + 5, 48)
```

```
旧城派出走不是硬失败。
```

## 十、旧城派最终结算  
**10.1 触发条件**  
最终结算触发：  
```
旧城派倒计时结束。
或
玩家选择“不再阻拦”。
或
第 48 天结束前仍未 resolved。
或
旧城派人数被压到低阶阈值以下并满足结算条件。

```
第 49 天前规则：  
```
第 49 天前，旧城派必须进入最终结算或被标记为已压制。
第七霜落期间不再开启新的旧城派阶段。
第七霜落期间不再开启新的离城倒计时。

```
**10.2 结算结果阈值**  
代码窗暂定实现值 / 测试窗必测：  
```
散去：
old_city_faction_count < old_city_low_threshold

部分出走：
old_city_low_threshold <= old_city_faction_count < old_city_high_threshold

大规模出走：
old_city_faction_count >= old_city_high_threshold

```
**10.3 旧城派散去**  
结果：  
```
old_city_faction_result = "scattered"
old_city_faction_count = 0
old_city_faction_resolved = true
trust_change = +3
panic_change = -2
population_leave = 0

```
终局标签候选：  
```
炉城裂缝被压住

```
**10.4 部分出走**  
代码窗暂定实现值 / 测试窗必测：  
```
partial_exodus_leave_count = min(
  floor(old_city_faction_count * 0.40),
  floor(current_alive_population * 0.12)
)

```
直接效果：  
```
trust_change = -5
panic_change = +5
old_city_faction_resolved = true
old_city_faction_result = "partial_exodus"

```
携带物资：  
```
cooked_food_loss = min(cooked_food_stock, partial_exodus_leave_count * 1)
coal_loss = min(coal_stock, partial_exodus_leave_count * 2)
wood_loss = min(wood_stock, partial_exodus_leave_count * 1)
steel_loss = min(steel_stock, floor(partial_exodus_leave_count / 2))

```
人口扣除：  
```
从总人口中扣除 leave_count。
不逐个追踪旧城派身份。
工程师最后扣除。
普通难度保留最低工程师保护线。

```
工程师保护线：  
```
若当前工程师 >= 2，则出走后工程师不得低于 2。
若当前工程师 < 2，则不得因出走再扣工程师。

```
边界：  
```
部分出走不得直接导致关键系统立即不可运行。
但会造成劳动力、住房、食物、医疗与终局评价影响。
旧城派出走不是死亡，不进入遗体系统。

```
**10.5 大规模出走**  
代码窗暂定实现值 / 测试窗必测：  
