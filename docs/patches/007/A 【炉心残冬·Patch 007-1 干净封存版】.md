# A 【炉心残冬·Patch 007-1 干净封存版】  
主题：事件系统总规则、旧城派事件、固定增员事件 状态：可替换正文 归属：007 事件系统 覆盖范围：007-A1 / 007-A2 / 007-A3  
   
⸻  
   
## 1. 本 Patch 定位  
007 是《炉心残冬》的事件系统。  
它负责把城市状态转化为玩家能看见、能回应、能承诺、能失败的事件。  
本 Patch 处理：  
```
事件系统总规则
事件触发与优先级
事件数量限制
承诺任务基础规则
事件被压下后的处理
旧城派事件系统
固定增员事件系统
007 与 006C 路线手段的接口
007 与日结保护的关系

```
本 Patch 不处理：  
```
具体信任数值
具体恐慌数值
具体食物 / 煤炭 / 医疗数值
旧城派具体人数阈值
固定增员具体人数比例
终局评价权重
第七霜落具体伤害数值

```
未封存数值不得由代码窗猜测。  
   
⸻  
   
## 2. 事件系统目标  
事件系统的目标不是随机惩罚玩家，而是让城市状态被玩家看见。  
事件应当做到：  
```
提示玩家当前城市问题。
给玩家一次选择或承诺机会。
把长期压力转化为阶段性危机。
避免后台状态无提示爆炸。
避免 AI 玩家不知道为什么失败。

```
事件系统不负责替玩家自动解决问题。  
事件可以：  
```
提示风险。
提供选项。
生成承诺任务。
结算一次性后果。
推进事件链。
触发 status 预警。

```
事件不能：  
```
偷偷替玩家执行命令。
偷偷修改持续状态。
绕过日结保护。
在后台结算未显示事件。
直接删除玩家已有系统状态。
凭空生成资源。

```
   
⸻  
   
## 3. 事件类型  
007 事件分为：  
```
重大事件
普通事件
固定日期事件
状态触发事件
承诺到期结算
非阻塞 status 提示
隐藏成就提示

```
说明：  
```
承诺到期结算独立于事件队列。
隐藏成就提示不进入事件队列。
非阻塞 status 提示不占事件名额。

```
   
⸻  
   
## 4. 每日事件执行顺序  
V1 采用回合制日结。  
每日推荐顺序：  
```
新一天开始
→ 先结算昨日到期承诺成功 / 失败
→ 移除已完成或失败的承诺任务
→ 释放承诺槽
→ 检查固定日期事件
→ 检查重大事件
→ 检查普通事件
→ 按事件优先级与数量限制显示事件
→ 显示 status / 非阻塞提示
→ 玩家进行当日操作
→ end_day
→ 日结
→ 进入下一天

```
说明：  
```
承诺到期结算是新一天开始时的系统批处理。
它不是当天事件队列的一部分。
它不占重大事件名额。
它不占普通事件名额。

```
   
⸻  
   
## 5. 事件优先级  
事件优先级方向为：  
```
1. 失败 / 危机倒计时事件
2. 第七霜落相关固定节点
3. 旧城派主线事件
4. 固定增员事件
5. 核心系统危机事件
6. 基础民生危机事件
7. 普通状态触发事件
8. 氛围提示 / 非阻塞提示

```
承诺到期结算不列入事件优先级表。  
承诺到期结算固定发生在事件检查前：  
```
新一天开始 → 承诺结算 → 当天事件优先级判断

```
   
⸻  
   
## 6. 同日事件数量限制  
同一天事件数量限制为：  
```
若当天有重大事件：
最多显示 1 个重大事件 + 1 个普通事件。

若当天无重大事件：
最多显示 2 个普通事件。

```
不占事件名额：  
```
承诺到期结算
隐藏成就
非阻塞 status 提示
第 48 天霜落前夜氛围提示

```
作废旧口径：  
```
同一日最多 1 个重大事件 + 2 个普通事件。
承诺到期结算占事件名额。
隐藏成就占事件名额。

```
   
⸻  
   
## 7. 重大事件定义  
重大事件是必须处理后才能 end_day 的事件。  
重大事件包括：  
```
失败 / 危机倒计时事件
第七霜落固定预警节点
旧城派主线关键阶段
固定增员事件
炉心红线
信任裂缝
炉城骚动
严重寒冷 / 医疗 / 死亡危机升级事件

```
重大事件特征：  
```
会影响城市主线走向。
会影响失败风险。
会影响人口、路线、终局或重大危机。
必须给玩家明确选择或确认。

```
重大事件未处理时，不允许直接 end_day。  
   
⸻  
   
## 8. 普通事件定义  
普通事件是提示型或中低压力事件。  
普通事件可以：  
```
提示民生压力。
提供一次性处理选项。
提供承诺任务入口。
给玩家提前预警。

```
普通事件可以被更高优先级事件压下。  
普通事件被压下时：  
```
不弹窗。
不结算选项。
不偷偷替玩家选择。
不偷偷生成承诺。
不偷偷改变资源 / 信任 / 恐慌。

```
若问题仍存在，次日重新检查或转为 status 预警。  
   
⸻  
   
## 9. 事件被压下后的处理  
若某事件因高优先级事件或数量限制被压下：  
```
不弹完整事件。
不结算事件选项。
不偷偷替玩家选择选项。
不偷偷生成承诺。
不偷偷触发承诺失败。
不偷偷消耗资源。
不偷偷改变信任 / 恐慌。

```
后续处理：  
```
若次日问题消失，事件取消。
若次日问题仍存在，重新检查触发。
若问题升级，触发更高阶段事件。
若暂不弹窗，则至少转为 status 预警。

```
说明：  
```
事件被压下不等于事件已经发生。
没有展示、没有选择，就不结算选项后果。

```
   
⸻  
   
## 10. 事件冷却  
事件冷却只限制同类事件重复弹窗。  
事件冷却不代表：  
```
问题消失。
惩罚暂停。
日结停止。
系统风险冻结。
上一次事件效果持续。

```
规则：  
```
同类事件冷却中，不重复弹同类普通事件。
对应问题仍按日结系统继续恶化或缓和。
若问题达到更高阶段，可以触发升级事件。
status 仍应提示持续存在的问题。

```
   
⸻  
   
## 11. 事件显示格式  
事件显示应包含：  
```
事件标题
事件类型
触发原因
当前状态摘要
可选选项
不可用额外手段
即时后果方向
承诺任务提示
冷却或前置限制

```
事件选项必须让玩家知道大致后果方向。  
允许显示：  
```
信任可能下降
恐慌可能上升
消耗熟食
需要建筑运行
需要路线解锁
生成承诺任务

```
禁止：  
```
完全隐藏重大后果。
把不可用选项塞进 A / B / C。
让玩家误以为事件已经自动执行命令。

```
   
⸻  
   
## 12. 可用 / 不可用选项规则  
正式选项编号只包含当前可执行选项。  
规则：  
```
可用选项进入 A / B / C。
不可用选项不进入正式编号。
不可用选项可显示在“当前不可用的额外手段”中。
不可用原因必须写清。
玩家强行输入不可用选项时，返回失败原因。

```
不可用选项失败时：  
```
不改变状态。
不结算效果。
不结算代价。
不消耗冷却。
不生成承诺。

```
示例：  
```
【当前不可用的额外手段】
- 共食誓餐：需要守炉堂已建成并运行，且熟食足够。
- 登记审查：需要旧城派危机已激活，且巡查所运行。

```
   
⸻  
   
## 13. 事件与核心命令边界  
事件不得永久自动执行核心命令。  
核心命令包括但不限于：  
```
furnace
overload
woodfuel
ration
workers
heat

```
事件可以：  
```
提示玩家可执行命令。
说明执行命令的可能后果。
提供一次性临时事件效果。

```
事件不能：  
```
永久修改核心命令状态。
替玩家输入命令。
让玩家误以为命令状态已经改变。

```
说明：  
```
事件负责让问题被看见。
命令负责真正改变城市运行状态。

```
   
⸻  
   
## 14. 承诺任务基础规则  
承诺任务是玩家在事件中选择“我会在期限内解决”的结果。  
承诺任务不等于持续增益。  
承诺任务不会：  
```
每天提高信任。
每天降低恐慌。
暂停原本的日结惩罚。
自动生成资源、住房、医疗容量或煤炭。
自动修复旧城派、医疗、食物、住房或炉心压力问题。

```
承诺任务只提供：  
```
一个限时目标。
一个成功 / 失败检查。
一次成功奖励或失败惩罚。

```
   
⸻  
   
## 15. 承诺任务结算时点  
承诺任务统一在新一天开始时结算。  
顺序：  
```
新一天开始
→ 检查 active 承诺任务
→ 若目标已达成，结算成功
→ 若 current_day > promise_deadline_day 且目标未达成，结算失败
→ 移除已结算承诺
→ 释放承诺槽
→ 再检查当天事件触发

```
禁止：  
```
在 end_day 中途结算承诺。
end_day 结算一次，新一天开始再结算一次。
承诺到期后同时触发多次成功 / 失败。

```
   
⸻  
   
## 16. 承诺截止日含义  
promise_deadline_day 表示最后一个仍可完成目标的日期。  
规则：  
```
玩家可以在 promise_deadline_day 当天继续操作。
promise_deadline_day 当天 end_day 后，系统进入下一天。
下一天开始时，按上一日日结后的最终状态检查承诺是否完成。

```
示例：  
```
第 20 天开启承诺。
promise_deadline_day = 23。

玩家可在第 20、21、22、23 天完成目标。
第 23 天 end_day 后完成日结。
第 24 天开始时检查承诺目标。

```
   
⸻  
   
## 17. 承诺任务上限  
全局承诺任务上限：  
```
同一时间最多存在 2 个 active 承诺任务。

```
同类承诺上限：  
```
同一时间最多存在 1 个同类承诺任务。

```
承诺槽满时：  
```
承诺类选项不可用。
不可用原因必须显示。
不可用选项不得进入正式 A / B / C 编号。

```
V1 默认不提供无代价取消承诺按钮。  
若后续提供取消功能：  
```
取消视为主动放弃，并立即按失败结算。

```
   
⸻  
   
## 18. 承诺成功与失败  
承诺可提前完成。  
规则：  
```
如果承诺目标提前达成，新一天开始时即可结算成功。
成功后移除承诺任务，并释放承诺槽。

```
承诺成功可以：  
```
提高信任。
降低恐慌。
降低旧城派人数。
降低对应风险。
终止对应承诺链。

```
承诺失败可以：  
```
降低信任。
提高恐慌。
提高旧城派人数。
提高危机风险。
推动事件链进入下一阶段。
让对应问题更容易再次触发。

```
承诺失败不得：  
```
直接秒杀玩家。
直接造成大量死亡。
直接触发旧城派最终出走。
直接扣除出走人口。
直接提前第七霜落。
直接触发炉心爆炸。
直接绕过危机倒计时。

```
   
⸻  
   
## 19. 连锁升级事件  
连锁升级用于处理长期未解决的问题。  
V1 连锁升级最多 3 层：  
```
第 1 层：status 预警
第 2 层：事件 / 承诺 / 选择
第 3 层：危机或严重后果

```
规则：  
```
同一问题最多升级 3 层。
第 3 层之后不再无限升级。
连锁升级不直接凭空生成资源。
连锁升级不直接跳过已有死亡、医疗、食物或住房系统。

```
旧城派使用自身阶段结构，不叠普通 1～3 层连锁升级。  
   
⸻  
   
## 20. 007 与 006C 路线手段的关系  
007 事件可以显示 006C 路线手段，但必须按 006C 已封存条件判断。  
誓言路线手段：  
```
守炉宣誓
敲响悼亡钟
共食誓餐
留城劝诫

```
铁腕路线手段：  
```
巡查
发布公告
临时留置
登记审查

```
可用条件：  
```
对应路线已选择。
对应炉律已签署。
对应核心建筑已建成。
对应核心建筑处于运行状态。
对应主动行动不在冷却中。
满足行动自身目标条件。

```
不可用时：  
```
可显示在“当前不可用的额外手段”中。
不得进入正式 A / B / C 选项。
玩家强行输入时，系统返回失败原因。
不改变状态。
不结算效果。
不消耗冷却。

```
   
⸻  
   
## 旧城派事件系统  
## 21. 旧城派定位  
旧城派是中期主线危机系统。  
它代表一部分居民相信南方还有另一座城，开始动摇是否继续留在炉城。  
旧城派不是普通随机事件。  
旧城派不属于 006C 誓言 / 铁腕路线。  
006C 只提供接口手段：  
```
誓言路线：留城劝诫
铁腕路线：登记审查

```
   
⸻  
   
## 22. 旧城派固定触发日  
旧城派固定在第 24 天触发。  
触发事件：  
```
南方来信

```
作废旧口径：  
```
旧城派在第 22～28 天之间浮动触发。

```
第 24 天触发理由：  
```
玩家已经经历早期求生。
社会适配和工程发展已有一定基础。
固定触发便于事件表、天气表、第七霜落预警和 AI 玩家规划对齐。

```
   
⸻  
   
## 23. 旧城派数据字段  
旧城派使用人数池。  
推荐字段：  
```
old_city_faction_active: bool
old_city_faction_count: int
old_city_faction_stage: int
old_city_faction_countdown_day: int | null
old_city_faction_resolved: bool
old_city_faction_result: str | null

```
旧城派人数代表：  
```
已经相信南烬城传言，并有离城倾向的人数。

```
旧城派人数可以：  
```
增长
降低
进入倒计时
最终散去
部分出走
大规模出走

```
   
⸻  
   
## 24. 旧城派人数增长原则  
旧城派人数受城市长期状态影响。  
影响方向包括：  
```
信任低
恐慌高
长期饥饿
长期寒冷
医疗崩溃
大量死亡
遗体长期未处理
固定增员拒绝或部分接纳
承诺失败
旧城派事件选项

```
终火誓约 / 最高戒令生效后：  
```
信任 / 恐慌失败轴可能被改写。
但信任 / 恐慌数值仍继续存在。
信任 / 恐慌仍继续参与旧城派人数计算。

```
说明：  
```
失败轴改写不是删除城市情绪。
居民不因为系统不判负就自动愿意留下。

```
   
⸻  
   
## 25. 旧城派阶段  
旧城派阶段结构：  
```
阶段 1：南方来信
阶段 2：暗中传言
阶段 3：公开集结
阶段 4：离城倒计时
阶段 5：最终结算

```
旧城派每个阶段事件每局只触发一次。  
旧城派必须在第七霜落前完成主要结算。  
规则：  
```
第 49 天前，旧城派必须进入最终结算或被标记为已压制。
第七霜落期间不再开启新的旧城派阶段。
第七霜落期间不再开启新的离城倒计时。

```
   
⸻  
   
## 26. 阶段 1：南方来信  
触发日：  
```
第 24 天

```
事件作用：  
```
告知玩家炉城内部出现“南烬城”传言。
初始化旧城派人数池。
开启旧城派系统。

```
选项方向：  
```
A：公布来信
B：压下来信

```
后果方向：  
```
公布来信：
信任可能小幅上升或保持。
恐慌可能小幅上升。
旧城派初始人数小幅增加。

压下来信：
信任可能下降。
恐慌可能维持或小幅上升。
旧城派初始人数正常生成，并进入地下传言状态。

```
   
⸻  
   
## 27. 阶段 2：暗中传言  
触发方向：  
```
旧城派人数达到低阶阈值。
或南方来信后，旧城派传言持续增长。

```
选项方向：  
```
A：公开解释
B：暂不处理

```
后果方向：  
```
公开解释：
旧城派人数小幅下降，或降低后续增长趋势。

暂不处理：
不消耗资源。
旧城派人数继续增长。

```
阶段 2 不触发离城倒计时。  
   
⸻  
   
## 28. 阶段 3：公开集结  
触发方向：  
```
旧城派人数达到中阶阈值。

```
该阶段开始显示旧城派人数和趋势。  
显示方向：  
```
旧城派人数：X
趋势：增长 / 持平 / 下降

```
普通事件选项：  
```
A：公开说明
B：加强巡查
C：暂不处理

```
后果方向：  
```
公开说明：
旧城派人数小幅下降。


```
```
加强巡查：
旧城派人数小幅下降。
可能降低信任或提高恐慌。

```
```

暂不处理：
旧城派人数继续增长。

```
说明：  
```
公开说明只是普通沟通手段，不要求誓言路线。
加强巡查只是普通秩序维护，不要求铁腕路线。
选择公开说明不会锁定誓言路线。
选择加强巡查不会锁定铁腕路线。

```
006C 路线手段若可用，应单独显示为额外手段，不与普通选项混淆。

## 29. 阶段 4：离城倒计时  
触发方向：  
```
旧城派人数达到高阶阈值。

```
事件作用：  
```
提示玩家在截止日前压低旧城派人数。
若截止日前无法压低旧城派人数，一部分人将离开。

```
选项方向：  
```
A：承诺压低旧城派人数
B：争取时间
C：不再阻拦

```
后果方向：  
```
承诺压低旧城派人数：
截止日前若成功，信任上升，恐慌下降，旧城派人数下降。
截止日前若失败，信任下降，恐慌上升，旧城派人数增加。
不会提前触发最终出走结算。

争取时间：
旧城派人数不变或小幅增加。
可能延后部分压力，但不解决问题。

不再阻拦：
根据旧城派人数产生散去 / 部分出走 / 大规模出走结果。

```
   
⸻  
   
## 30. 旧城派承诺失败护栏

旧城派承诺失败不得：  
```
提前触发离城倒计时。
跳过离城倒计时。
立即触发最终出走结算。
直接扣人口。
直接造成死亡。

```
说明：  
```
旧城派承诺失败只让局势更糟，不直接跳过主线阶段。

```

## 固定增员事件系统  
## 34. 固定增员总原则  
固定增员是固定日期事件，不是随机事件。  
固定增员事件均为重大事件。  
规则：  
```
固定增员事件每局各触发一次。
固定增员事件不会被普通民生事件覆盖。
固定增员事件必须处理后才能 end_day。

```
固定增员用于制造人口压力、劳动力机会和道德选择。  
   
⸻  
   
## 35. 固定触发日  
固定增员共三批：  
```
第 6 天：早期求生者
第 19 天：中期工程残队
第 37 天：后期难民潮

```
说明：  
```
第 6 天：玩家已完成基础建造与食物 / 住房初步理解，适合加入小规模压力。
第 19 天：寒潮扩张阶段，城市开始需要更多工程与生产能力。
第 37 天：深冬末期、黑霜前夕，适合制造终局前最大安置压力。

```
   
⸻  
   
## 36. 接纳选项总规则  
固定增员事件默认提供三类选项：  
```
全部接纳
部分接纳
拒绝接纳

```
V1 不允许玩家手动输入接纳人数。  
部分接纳由系统预设比例。  
具体人数与比例由后续数值表确认。  
   
⸻  
   
## 37. 新增人口状态  
新增人口可以包含：  
```
健康工人
工程师
儿童
患病人口
重症人口
伤残人口

```
新增人口状态必须立即进入对应人口池。  
固定增员处理后：  
```
新增人口立即加入总人口。
健康工人 / 工程师从当天起计入可工作人口。
患病、重症、伤残人口立即计入对应人口池。
儿童立即计入儿童人口。
新增人口立即参与当日住房、食物和医疗压力。

```
但：  
```
新增健康人口不自动分配岗位。
新增人口不自动进入 workers。
玩家需要手动调整工作分配。

```
   
⸻  
   
## 38. 新人口生效时机  
固定增员事件处理后，人口立即生效。  
规则：  
```
开门当天，人已经进城。
当天就要吃饭。
当天就占房。
当天就占病床。
当天就计入疾病压力。

```
end_day 前必须刷新风险预览。  
预览至少包括：  
```
住房压力
食物压力
医疗压力
疾病压力
无住所风险
信任 / 恐慌风险
旧城派风险

```
   
⸻
