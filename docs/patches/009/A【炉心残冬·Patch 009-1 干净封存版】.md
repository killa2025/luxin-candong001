主题：终局报告结构、通用文案池与叙事式炉城报告 状态：可替换正文 归属：009 结局系统 覆盖范围：009-A / 009-A 小修补丁 r1

## 1. 本 Patch 定位  
009-1 负责确定《炉心残冬》V1 的终局报告结构、通用结局文案池、玩家主动结束文案池、叙事式炉城报告与执政官拷问文案。  
本 Patch 处理：  
```
终局报告文字结构
通用硬失败文案池
信任失败 / 恐慌失败修正口径
玩家主动结束文案池
第七霜落存活结局文案池
高质量胜利 / 标准胜利加重版文案
崩坏幸存 / 残火未灭通用文案方向
叙事式炉城报告
重大状态附加句文案池
执政官拷问文案池
玩家可见日期口径
009-B / 009-C 接口占位

```
本 Patch 不处理：  
```
完整标签筛选代码
重大决定权重
路线结局判定
炉律专属完整结局文案
誓言 / 铁腕专属长文案

法典路线定义
炉人路线定义
事件系统重写
普通日结事件文案
AI 玩家汇报
自动存档
UI 代码结构

```
说明：  
```
009-1 负责“终局怎么写”。
009-2 负责“哪些标签被选中、哪些附加句出现、重大决定如何进入终局报告”。
009-C 若后续存在，负责“炉律 / 路线专属完整结局文案”。

```

## 2. 名词口径  
009-1 统一以下称呼：  

| 对象   | 玩家可见称呼        |
| ---- | ------------- |
| 玩家   | 执政官           |
| 城市居民 | 炉人 / 守炉人 / 居民 |
| 城市   | 炉城            |
| 终局灾害 | 第七霜落          |
| 供暖核心 | 炉心            |
  
规则：  
```
执政官 = 玩家 / 决策者。
炉人 / 守炉人 / 居民 = 城市居民。
炉城 = 城市本体。
终局审判针对执政官，不针对普通居民。

```
应写：  
```
执政官签署了誓约。
居民在炉心旁留下了名字。
炉人没有离开，但他们也没有忘记。

```
不应写：  
```
守炉人决定了城市的命运。
炉人签署了最高命令。

```
   
⸻  
   
## 3. 文案总风格  
009-1 终局文案采用：  
```
冷峻
克制
短句
少解释
少煽情
不替玩家辩护
不替系统宣判道德答案

```
文案应像：  
```
一份炉城档案。
一段灾后记录。
一场没有法槌的审问。

```
不应像：  
```
系统热血表彰。
纯粹胜利颂歌。
大段道德说教。
客服式结算提示。

```
核心句式方向：  
```
城市活下来了，但……
炉心还在烧，可……
人们没有离开，只是……
执政官做出了选择，炉城替他保存了后果。

```
   
⸻  
   
## 4. 玩家可见日期口径  
玩家可见终局文案中尽量不使用后台写法：  
```
D55
D49
D49～D55

```
改用自然文本：  
```
第 55 天
第七霜落开始时
第七霜落结束时
最后七天
最终
霜落前夜

```
示例：  
```
错误：D55 之后，炉心仍在燃烧。
正确：第七霜落结束后，炉心仍在燃烧。

错误：D49～D55 死亡人数。
正确：最后七天里死去的人。

```
规则：  
```
后台字段可以继续使用 d49 / d55 / frostfall_days。
玩家可见文案尽量使用自然中文。

```

## 终局报告结构  
## 5. 终局报告总结构  
第 55 天 end_day 后，若进入终局报告，推荐结构如下：  
```
一、终局标题
二、主文案
三、叙事式炉城报告
四、重大状态附加句
五、执政官拷问文案
六、009-B / 009-C 标签与路线接口

```
终局报告不再默认展示大段数值摘要表。  
作废旧结构：  
```
终局摘要表
大段数值表
以 D55 / D49～D55 作为玩家主文案标题

```
保留：  
```
后台仍可读取数值。
UI 可在详情页展示数值。
主报告以叙事式文本为主。

```

## 7. 存活结局模板  
若未触发系统硬失败，且第七霜落结束后仍有存活人口，则进入存活结局模板。  
```
【终局标题】

第七霜落结束了。

【生存状态主文案】

【叙事式炉城报告】

【重大状态附加句，0～3 条】

【执政官拷问文案，1 条】

```
说明：  
```
存活结局可以很沉重。
存活不等于高质量胜利。
崩坏幸存 / 残火未灭仍然使用存活结局模板。

```

系统硬失败使用：  
```
【失败标题】

【失败主文案】

【一句式失败原因】

【硬失败通用结尾】

```
系统硬失败只对应：  
```
population_zero：人口归零。
core_collapse：炉心崩毁。
trust_exile：信任归零，且未被终火誓约承接，执政官被流放。
panic_expelled：恐慌达到失败条件，且未被最高戒令压制，执政官被驱逐。

```
玩家主动结束不放入系统硬失败模板。

```
最后一份名册没有被收回。

炉心仍在夜里发出低声的震动，但已经没有人需要它。
床铺空了。
食堂空了。
医疗站也空了。

炉城没有倒塌。
只是再也没有居民。

第七霜落没有带走炉城的墙。
它带走了墙里所有会说话的人。

最后的炉火照亮了空屋。
执政官的命令仍能被写下。

```

```
炉心没有熄灭。
```

## 12. 信任失败 / 执政官被流放

```
执政官被流放

```

```
信任失败不是炉城立即毁灭。
信任失败是居民不再承认执政官继续执政。
这是玩家失败，不等于炉城必然当场灭亡。

```
主文案池：  
```
居民没有冲进执政厅。

他们只是停下了。

停下工作。
停下等待。
停下相信下一道命令会把他们带到明天。

炉心还在烧。
但执政官已经失去了让人靠近它的理由。

最后一份公告贴在炉心旁。
没有人撕下它。
也没有人再读它。

信任不是被砸碎的。
它是一层一层冻住的，直到再也无法融开。

炉城仍有煤。
仍有墙。
仍有名字。

但居民不再相信执政官能带他们活下去。

命令还在。
城市已经不再回应。

最后，炉城打开了门。

不是为了迎接谁。
而是为了让执政官离开。

```
一句式原因：  
```
居民不再相信你有资格继续带领他们。

```

```
这是玩家失败，不等于炉城必然当场灭亡。
```

```
恐慌没有形状。

炉城不是被风雪冲散的。
它是从内部散开的。

没有人再等待日结。
没有人再相信下一道命令会比尖叫更快。

有人抢煤。
有人砸门。

最后，人们聚到炉心前。
他们不再等待执政官开口。
```

## 14. 硬失败通用结尾  
系统硬失败文案结束后，可接以下收束句之一。  
```
你失败了。

这个炉城也许会迎来新的长官，也许就此陨落。
但这一切都已经与你无关了。

你可以选择重新开始，也可以就此放弃。
选择权在你，执政官。
本局结束。
炉城没有再听见你的命令。

它也许会继续挣扎，也许会在风雪里彻底沉下去。

那是另一个故事。
而你已经被写出这份档案。

你的执政到此为止。
炉心是否还能烧到明天，居民是否还能熬过下一夜，已经不再由你决定。

风雪收下了你的失败。
炉城收回了你的名字。

```
规则：  
```
硬失败结尾不强行断言炉城一定灭亡。
硬失败结尾保留玩家重新开始 / 放弃的选择感。
硬失败结尾不羞辱玩家。

```

## 玩家主动结束文案池  
## 15. 玩家主动结束口径  
玩家主动结束不是系统硬失败。  
口径：  
```
玩家主动结束 = 执政官亲手封存本局。
系统不替玩家判负。
风雪不替玩家判负。
城市状态按 player_ended 记录。

```
标题：  
```
执政官终止执政

```
主文案池：  
```
执政官终止了本局。

炉心仍在燃烧。
名册仍未写完。
有些门还关着，有些床还空着。

这不是系统宣判的失败。
这是执政官亲手合上的档案。

本局在执政官的命令下结束。

城市没有替你做决定。
风雪也没有。

结束是一个选择。
和所有选择一样，它会留下记录。

```
一句式封存状态：  
```
本局由执政官亲手封存。

```
收束句：  
```
炉城档案封存。

不是因为所有问题都有了答案。
而是因为执政官决定，不再继续追问。

```

```
炉心仍在燃烧。
食堂重新排起队。
医疗站的灯没有熄。
清晨点名时，大多数名字仍有人回答。

这是一场少见的胜利。

他们没有走到最终。
却把空出来的位置，留给了后来的人。

风雪退去时，炉城没有欢呼。

人们只是走出门，确认烟囱还在，名册还在，孩子还在。
有人把冻硬的门闩重新推开。
有人开始计算下一顿饭。

胜利没有声音。
它只是让城市还能继续做明天的事。

煤仓没有彻底见底。
病床没有完全溢出。

大多数人仍在自己的床上醒来。
```

```
墓园 / 冷藏坑 / 余烬名册上，已经留下 {total_deaths} 个名字。
最后七天里，第七霜落又带走了 {frostfall_deaths} 个再过几天就能见到曙光的人。

{illness_sentence}

{trust_panic_sentence}

{core_sentence}

{coal_food_sentence}

{hope_sentence}

```
规则：  
```
叙事式报告可以嵌入少量数字。
不展示完整表格。
不在主报告中堆六大系统分数表。
详细数值可放到详情页或调试面板。
```

## 重大状态附加句  
## 27. 重大状态附加句定位  
重大状态附加句用于补充终局主文案。  
它只记录：  
```
长期
反复
大规模
高比例
危机级
路线级
造成不可逆后果

```
不记录普通小事。  
默认不进入终局审判：  
```
死过 1 个人。
某天有人没被治疗。
有人露宿过 1 夜。
某天出现少量患病。
某天建筑短暂停工。
某天使用过 1 次 heat。
某天短暂食物不足。
某天信任或恐慌小幅波动。
某天煤炭紧张但没有造成连续崩坏。

```
说明：  
```
小事可以进入日结日志、事件日志、状态提醒。
小事不应挤占终局审判空间。

```

## 执政官拷问文案池  
## 34. 拷问文案定位  
执政官拷问文案用于终局最后一段。  
它不负责判定胜负。  
它负责提醒玩家：  
```
城市活下来了，不等于一切都值得。
城市失败了，也不等于每个选择都没有意义。

```
规则：  
```
拷问文案不羞辱玩家。
拷问文案不替系统宣判唯一道德答案。
拷问文案应短、冷、具体。

```
   
⸻  
   
## 35. 通用拷问文案池  
```
执政官，炉城活了下来。

但它不是自己活下来的。
它是被煤、命令、恐惧、忍耐和死亡一起推过来的。

现在，风停了。
你还愿意看一眼自己留下的名册吗？
你守住了炉心。

可炉心不是城市的全部。
那些没被炉火照到的人，也会进入档案。
你可以说这是必要代价。

炉城不会反驳。
它只会把代价逐条保存。
第七霜落结束后，命令仍然有效。

只是有些人已经不在命令能抵达的地方。
执政官，城市没有问你是否后悔。

它只问你：如果寒冬再来一次，你还会这样做吗？

```
   
⸻

## 防漏洞规则  
## 42. 防信任 / 恐慌失败写成炉城必灭  
信任失败 / 恐慌失败是玩家失败。  
规则：  
```
信任失败 = 执政官被流放。
恐慌失败 = 执政官被驱逐。
信任 / 恐慌失败不等于炉城立即灭亡。

```
禁止写：  
```
信任归零，炉城立刻毁灭。
恐慌达到上限，所有居民立刻死亡。
恐慌失败一定伴随旧城派出走。

```
   
⸻  
   
## 43. 防恐慌文案混入旧城派  
通用恐慌失败、恐慌崩坏、秩序崩坏文案中，不提：  
```
南方传言
南方
旧城派
旧城派出走

```
除非 009-B / 009-C 明确选择旧城派结局标签。  
理由：  
```
恐慌失败是社会失控。
旧城派是另一条事件 / 路线线索。
二者不能在通用恐慌文案里乱炖。

```
   
⸻  
   
## 44. 防胜利写得太轻  
高质量胜利、标准胜利都必须保持重量。  
禁止写成：  
```
大家欢呼起来。
炉城迎来完美结局。
所有人都原谅了执政官。
寒冬终于结束，一切都会变好。

```
应保持：  
```
胜利有代价。
标准胜利也有损失。
高质量胜利也不能删掉死者。
炉城活下来了，但档案不会替执政官美化所有选择。

```
   
⸻  
   
## 45. 防终局报告变成数值表  
终局主报告不展示大段数值表。  
允许：  
```
叙事式嵌入少量数字。
详情页展示数值。
调试面板展示完整字段。

```
禁止：  
```
主文案后直接贴六大系统完整表格。
主报告核心体验变成成绩单。
玩家先看分数，再看城市。

```
   
⸻  
   
## 46. 防小事挤占终局审判

禁止：  
```
死过 1 人就触发重大死亡审判。
露宿 1 夜就触发寒屋终局标签。
使用 1 次 heat 就触发救命热源标签。
短暂食物不足就触发饥荒城市文案。

```
这些筛选由 009-B 进一步收紧。
