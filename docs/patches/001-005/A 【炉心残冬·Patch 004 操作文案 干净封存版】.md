# A 【炉心残冬·Patch 004 操作文案 干净封存版】  
版本定位：命令系统、操作流程与面板氛围文案封存 用途：供后续伪源码、命令解析、操作反馈、AI 玩家接口继承  
   
⸻  
   
## 一、合并后的干净版正文  
**1. 本 Patch 只处理什么**  
Patch 004 只处理：  
* 玩家可用命令  
* 面板查看命令  
    * 建造、拆除与确认机制  
* 人手分配命令  
* 炉心、过载、应急加热与木材应急燃烧命令  
* 研究与城约命令入口  
* 结束一天命令  
* 无效命令处理  
* AI 玩家反馈格式  
* 主要面板氛围文案  
本 Patch 不重复处理：  
* 建筑占槽数值  
* 建筑建造上限  
* 人口状态链  
* 资源日结算细节  
* 每条城约具体效果  
* 每项科技具体数值  
* 天气表  
* 随机事件文本  
* 终局结算  
   
⸻  
   
**2. 命令系统核心原则**  
《炉心残冬》是文字策略生存游戏。 玩家通过输入命令查看局势、调整城市、推进日期。  
命令系统需要满足：  
* AI 玩家能稳定读取。  
* 人类旁观者能看懂进展。  
* 命令格式尽量简单。  
* 出错时必须给出原因。  
* 高风险操作必须给出警告。  
* 关键操作需要二次确认。  
* 每次操作后只显示变化项、影响与预警，不强制刷新完整面板。  
命令系统不是纯终端模拟，而是文字控制台。 它要让玩家知道自己做了什么，也知道这个选择会让谁付出代价。  
   
⸻  
   
**3. 面板氛围描述规则**  
主要面板打开时，系统应在数据前显示一段简短氛围描述。  
适用命令：  
* status  
* city  
* laws  
* tech  
* workers  
* end_day 结算前  
* end_day 结算后  
* 特定危机或特殊状态提示  
规则：  
* 常规面板每次 1～2 句。  
* 常规描述不超过 100 字。  
* 放在面板数据之前。  
* 描述寒冷、疲惫、抉择、人心与城市衰败。  
* 可以悲惨，但不要过长。  
* 不替代数值信息。  
* 不影响 AI 玩家读取数据。  
* 第七霜落临近属于终局特殊提示，可以略长。  
普通操作命令如 assign、build、heat 不强制添加氛围文案，避免操作反馈过长。  
   
⸻  
   
**4. 面板氛围文案**  
**status 面板**  
炉心井还在烧，但它的光根本照不到每一张脸。有人挤在热管旁发抖，有人在病床上叹息，而你的抉择将决定今天先救煤，还是先救人。  
**city 面板**  
城市像一圈圈冻裂的伤口，内环还残留炉心余温，外环却已被风雪啃得发白。每一座新建筑都像是在问：谁值得占据这仅剩的热量？  
**laws 面板**  
誓约碑前没有人高声争辩，只有冻僵的手印留在铁板边缘。每一条炉律都不是答案，只是把某些人的绝望与痛苦写得更像秩序。  
**tech 面板**  
研究所的灯亮到深夜，工程师们用冻裂的嘴唇低声争执。他们不是在追求进步，只是在和死亡讨价还价，试图把明天争回来一些。  
**workers 面板**  
分工牌上有些名字被划掉，又有新的名字挤上去；炉城已经没有多余的铁做新工牌。所以能站起来的人，哪怕还在咳血，也会被写进今日的岗位。  
**end_day 结算前**  
夜晚压过来，炉城的边界被风雪一点点抹掉。若现在结束这一天，所有没来得及处理的寒冷、饥饿和病痛，都会替你做出选择。  
**end_day 结算后**  
清晨没有真正到来，只是黑夜变薄了一点。炉心还亮着，但名单上少了几个人；活下来的人没有欢呼或哀悼的力气，只是继续排队领今天那一点微薄的热量。  
**狩猎区发现时**  
风雪里有一抹白影一闪而过，像冬天最后舍不得熄灭的火。猎人说，那是雪狐——皱巴巴、圆滚滚、还活着的白色小东西，也是这片冻土上最后一部分仍能奔跑的生命。  
它们可以成为食物，也可以成为火种。 只是炉城里的锅已经见底，而孩子们还在排队。  
**信任过低时**  
人们仍然听从命令，但不再抬头看你。炉心的光还在城中摇晃，可他们已经开始怀疑，自己究竟是在被守护，还是被消耗。  
**恐慌过高时**  
风雪还没有冲进城里，恐惧已经先压进人群。有人囤食，有人拒绝上工，有人在夜里高喊：如果明天注定要死，今天为什么还要服从？  
**煤炭不足时**  
煤堆低得像一句快要说完的谎言。炉心仍在索要燃料，而仓库里仅剩的黑色碎块，已经在给睡梦里的人设计墓碑。  
**食物不足时**  
食堂排队的人比昨天更安静。没人再抱怨份量，因为每个人都看见了锅底，也看见了排在自己身后的孩子。  
**医疗崩溃时**  
医疗站里没有足够的床，也没有足够的热。病人被分成“还能等”和“等不了”，而医生只能说服自己这种区别不是判决。  
**炉心关闭时**  
炉心沉默下去的那一刻，整座城都像屏住了呼吸。寒冷不会说话，它只是走进每一间屋子，而风声像死亡的悼词。  
**开启过载时**  
炉心发出近乎痛苦的轰鸣，像一颗被迫继续跳动的心脏。人们感到一丝久违的暖意，却没人知道这份暖意会烧掉多少明天。  
**启用木材应急燃烧时**  
木材被推进炉膛，火光短暂亮起来。有人看着那些本该变成房梁、病床和采集棚的木料化成灰，眼神木然得像已经没有力气判断这是不是正确。  
**无住所人口出现时**  
有些人没有床位，只能靠在炉心边等天亮。炉城仍说自己庇护着所有人，但风知道，今晚并不是每个人都有屋顶。  
**儿童保护不足时**  
孩子们被安置在离炉心最近的角落，却仍然有人冻醒。大人们把仅剩的布盖在他们身上，然后转身去冰冷的工作地继续拼命干活。  
**儿童辅工启用时**  
孩子们被领到工地边缘，手套比他们的手大，工具比他们的胳膊沉。有人说这不是残忍，是活下去；可当第一个孩子学会沉默地低头搬煤时，炉城就已经失去了一些东西。  
**学徒制度启用时**  
学堂的炉火很小，照不亮整间屋子，但孩子们仍围在图纸和工具旁学习。大人们说他们是在为明天准备，可每个人都知道，所谓明天，只是今天还没有吞掉的那一部分孩子。  
**墓园建成时**  
墓园并没有让死亡有意义，只是让死亡有了位置。活人能分到多一口食物，而炉城也终于有了一个地方，安放那些没撑到春天的人。  
**火化场建成时**  
火化场的烟比炉心更沉默。它节省了土地，也节省了仪式，只是没人愿意承认：效率有时闻起来像灰。  
**临时超仓时**  
仓库被塞得满满当当，门缝都挤着求生的重量。可这些物资不是富余，只是另一种混乱；城市已经连保存希望都显得狼狈。  
**第七霜落临近时**  
风声变了，像有什么巨大的东西正在雪幕后靠近。炉城里所有人都明白，之前的寒冷只是练习，真正的审判即将逼近。谁有资格活过明天？是谁规定的？谁会被寒冬巨兽无情吞没？又是谁允许这一切继续发生？炉心仍在燃烧，可人性、火种、希望这些词，已经轻得像雪。明天还会到来吗？如果会，那里还会有人吗？

**6. 建筑编号规则**  
所有可重复或可指定目标的建筑，建成后自动生成编号。  
示例：  
```
住宅#1
住宅#2
医疗站#1
研究所#1
温室#1
小仓库#1

```
编号用途：  
* 拆除建筑  
* 应急加热  
* 分配人手  
* 查看状态  
* 区分多个同名建筑  
若玩家输入的目标存在歧义，系统必须要求指定编号。  
示例：  
```
存在多个目标：
医疗站#1
医疗站#2

请指定编号。
例如：
heat 医疗站#1

```

**7. 建造命令**  
**build**  
建造建筑。  
格式：  
```
build <建筑名> <区域>

```
示例：  
```
build 住宅 内环
build 医疗站 内环
build 研究所 中环
build 温室 中环
build 狩猎小屋 外环
build 小仓库 外缘储放区

```
系统需要检查：  
* 建筑是否存在  
* 区域是否允许  
* 槽位是否足够  
* 建筑数量是否达到上限  
* 资源是否足够  
* 科技是否解锁  
* 资源点是否可用  
* 是否违反互斥条件  
**建造生效时间**  
建筑成功建造后：  
* 立刻扣除资源。  
* 立刻占用槽位。  
* 立刻出现在 city 面板。  
* 立刻生成建筑编号。  
但建筑需要满足运行条件后，才会在当日 end_day 结算中生效。  
例如：  
食堂当天建成后，若玩家当天分配工人且食堂未停工，则可在当天结算中加工熟食。  
设计原因：  
允许玩家前期通过“当天建食堂、当天派人”缓解食物压力，避免不合理饿死。  
成功反馈示例：  
```
执行结果：
已建造：食堂#1，位置：内环。

影响：
内环槽位：已占用 2 槽。
木材：减少。
钢铁：减少。

预警：
食堂需要分配工人才能加工熟食。

可选下一步：
assign 4 工人 to 食堂#1
city
status

```
失败反馈示例：  
```
无法建造：温室
原因：中环剩余槽位不足。温室需要 3 槽，当前仅剩 2 槽。

```

**8. 拆除命令**  
**demolish**  
拆除建筑。  
格式：  
```
demolish <建筑名或编号>

```
示例：  
```
demolish 住宅#3
demolish 小仓库#1

```
系统反馈：  
* 拆除建筑  
* 释放槽位  
* 返还资源  
* 是否影响住房、医疗、生产、仓储  
若拆除会导致明显风险，应提示并要求确认。  
示例：  
```
警告：拆除住宅#3 后，住房容量将低于总人口。
可能产生无住所人口。
确认请输入：confirm demolish 住宅#3

```

**9. 确认命令**  
**confirm**  
确认高风险操作。  
格式：  
```
confirm <原操作>

```
示例：  
```
confirm demolish 住宅#3
confirm furnace off
confirm end_day
confirm sign 儿童辅工

```
确认规则：  
* 高风险操作必须绑定原操作。  
* 玩家必须原样输入系统要求的确认命令。  
* 如果确认前输入了其他非查看命令，当前确认失效。  
* 系统应提示确认已失效，需要重新发起操作。  
设计原因：  
避免 AI 玩家隔了几轮后误确认旧操作。

**10. 人手分配命令**  
**assign**  
分配人手到建筑或工作区域。  
格式：  
```
assign <人数> <人口类型> to <目标>

```
示例：  
```
assign 10 工人 to 大型煤矿机#1
assign 5 工人 to 伐木场#1
assign 3 工程师 to 医疗站#1
assign 4 工程师 to 研究所#1
assign 6 工人 to 食堂#1

```
**assign 的含义**  
assign 表示“设置目标岗位人数”，不是在原有人数基础上追加。  
例如：  
```
assign 8 工人 to 大型煤矿机#1

```
含义是：  
大型煤矿机#1 当前工人数被设置为 8。  
如果玩家再次输入：  
```
assign 8 工人 to 大型煤矿机#1

```
人数仍为 8，不会变成 16。  
设计原因：  
避免 AI 玩家重复输入导致岗位人数叠加爆炸。  
系统需要检查：  
* 是否有足够可用人口  
* 人口类型是否能执行该岗位  
* 建筑或资源点是否存在  
* 建筑是否可运行  
* 是否超过岗位上限  
* 是否至少分配 1 人  
成功反馈示例：  
```
执行结果：
已设置：8 名工人 → 大型煤矿机#1。

影响：
标准人手：10
预计效率：80%

预警：
低温作业将增加患病和冻伤风险。

```
**unassign**  
撤回某个岗位人手。  
格式：  
```
unassign <人数/all> from <目标>

```
示例：  
```
unassign 4 from 伐木场#1
unassign all from 狩猎小屋#1

```
反馈示例：  
```
执行结果：
已撤回：4 名工人 ← 伐木场#1。

影响：
当前伐木场#1 人手：2/8
预计效率：25%

```

**13. 应急加热命令**  
**heat**  
对单个建筑开启当日应急加热。  
格式：  
```
heat <建筑名或编号>

```
示例：  
```
heat 医疗站#1
heat 温室#1
heat 儿童保护所#1

```
  
系统需要检查：  
* 目标建筑是否允许 heat  
* 今日加热次数是否剩余  
* 煤炭是否足够  
* 目标建筑是否已经温度足够  
* 当日该建筑是否已经使用过 heat  
  
说明：  
heat 命令开局可用，不需要先研究科技解锁。  
应急加热装置是 heat 强化科技，不是 heat 解锁科技。  
  
  
反馈示例：  
```
执行结果：
已开启应急加热：医疗站#1。

影响：
持续时间：今日。
今日剩余加热次数：1。

预警：
当日结束后，应急加热会自动关闭。

```
若不需要加热：  
```
无需加热：医疗站#1 当前温度已满足运行需求。
未消耗煤炭。
```

**19. 当日操作限制**  
同一天内，玩家可以：  
* 多次查看面板。  
* 多次调整人手。  
* 多次尝试建造。  
* 在结束当天前调整炉心档位。  
* 在结束当天前开启或关闭木材应急燃烧。  
以下操作存在限制：  
* 城约签署受冷却限制。  
* 紧急令受次数限制。  
* 应急加热受每日次数限制。  
* 研究同一时间只能进行一个。  
* 建筑建造受资源、槽位、上限和科技限制。  
* 炉心过载受压力与研究限制。  
每日结算以 confirm end_day 时的最终状态为准。

**27. Patch 004 总结**  
本 Patch 确定《炉心残冬》第一版命令系统方向：  
* 玩家通过文字命令操作炉城。  
* 基础面板包括 status、city、laws、tech、workers。  
* 主要面板打开时显示简短凄凉氛围文案。  
* 建造使用 build。  
* 建筑建成后立刻扣资源、占槽、编号，并出现在 city。  
* 建筑需要满足运行条件后，才能在当日结算中生效。  
* 所有可指定建筑自动生成编号。  
* 拆除使用 demolish，高风险拆除需要 confirm。  
* 确认命令必须绑定原操作。  
* 人手分配使用 assign / unassign。  
* assign 表示设置目标岗位人数，不是追加人数。  
* 炉心档位使用 furnace。  
* 过载使用 overload。  
* 应急加热使用 heat。  
* heat 命令开局可用，不需要科技解锁。  
* 应急加热装置是 heat 强化科技，不是 heat 解锁科技。  
* heat 当日生效，日结后自动关闭。  
* 木材应急燃烧使用 woodfuel。  
* woodfuel on 持续到玩家手动关闭或被系统规则强制中止。  
* 煤炭恢复后，woodfuel 不会自动关闭。  
* 研究使用 research。  
* 城约签署使用 sign。  
* 配给入口使用 ration。  
* 结束当天使用 end_day。  
* end_day 前必须显示未处理风险清单。  
* end_day 预警按信息提示 / 强警告 / 硬阻塞分级处理。  
* 高风险操作需要明确警告或二次确认。  
* 每次操作反馈应固定格式，便于 AI 玩家理解。  
* 无效命令必须说明原因，并给出可尝试命令。  
* 中文命令可以作为别名，但内部统一转换成标准命令。  
* 每日结算以 confirm end_day 时的最终状态为准。  
核心原则：  
命令系统不是为了像真正终端一样冷冰冰。 它的作用是让 AI 玩家能稳定操作，让人类旁观者能看懂局势，让每一步选择都知道自己在付什么代价。  
面板文案负责让炉城活起来。 命令反馈负责让玩家看清楚。 确认机制负责防止一次误操作把整座城推进雪里。
