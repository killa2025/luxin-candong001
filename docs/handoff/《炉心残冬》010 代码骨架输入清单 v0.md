## 一、本文件定位  
当前没有单独发现“Patch 010 干净封存版”作为正式源文件。 因此本文件不是伪装成 010 原文，而是基于当前已封存主干整理出的：  
```
代码骨架输入清单

```
它解决的问题是：  
1. 代码窗应该接哪些系统。  
2. 各系统读取哪些字段。

## 二、代码窗总原则  
## 2.1 只接正式口径  
代码窗只能接：  
1. 已封存数值。  
2. 用户明确确认的覆盖规则。  
3. 已收敛的全局修正口径。  
4. 明确标记为可进代码骨架的机制。  
代码窗不得接：  
1. 待确认数值。  
2. 草案建议。  
3. 数值候选附录中未封存项。  
4. 测试窗分类。  
5. 文案方向推断出的机制。  
6. 旧口径。  
7. 被后续覆盖的原文。  
   
⸻  
   
## 2.2 PENDING 不得猜  
以下统一写 PENDING，不得补：  
```
PENDING_NUMERIC
PENDING_SOURCE
PENDING_RULE
PENDING_BALANCE
PENDING_UI
PENDING_EVENT
PENDING_LATER

```
代码窗可以建立字段，但不能填最终值。  
例如：  
```
woodfuel_conversion_ratio = PENDING_NUMERIC
children_lost_threshold = PENDING_NUMERIC
old_city_initial_count = PENDING_NUMERIC
tech_research_days = PENDING_NUMERIC

```
不得写成：  
```
代码窗先随便填个差不多的。

```
   
⸻  
   
## 2.3 游戏类型  
《炉心残冬》是：  
```
文字策略生存游戏
每日回合制 / 日计划制

```
核心原则：  
1. 玩家白天提交计划。  
2. end_day 后统一结算。  
3. 日结中途不能补煤。  
4. 日结中途不能补食物。  
5. 日结中途不能改岗位。  
6. 日结中途不能临时 heat。  
7. 日结中途不能临时 woodfuel。  
8. 日结中途不能签炉律。  
9. 日结中途不能研究科技。  
10. 所有严重后果必须在 end_day 前尽量预警。

## 五、玩家命令表  
## 5.1 面板命令  
```
status
city
laws
tech
workers
events
promises
save
autosave
final_report
help

```
最低用途：  
1. status：总览、风险、当前日、资源、人口、信任 / 恐慌。  
2. city：建筑、区域、槽位、运行状态。  
3. laws：普通适配城约、誓言与铁腕炉约。  
4. tech：工程科技树。  
5. workers：人手分配。  
6. events：今日事件 / 历史事件。  
7. promises：当前承诺任务。  
8. final_report：终局后查看报告。  
9. help：命令帮助。  
10. save：手动保存当前 GameState。  
11. autosave：查看最近一次 end_day 自动保存状态。  
  
5.1.1 保存与自动保存  
  
保存相关命令：  
  
save  
autosave  
  
规则：  
  
1. save 用于玩家手动保存当前 GameState。  
2. autosave 用于查看最近一次 end_day 自动保存状态。  
3. 每次成功完成 end_day 日结后，系统必须自动保存。  
4. 自动保存不需要玩家手动确认。  
5. 自动保存不替代手动保存。  
6. 自动保存至少保留 1 个固定槽位：autosave_end_day。  
  
字段建议：  
  
autosave_enabled = true  
last_autosave_day  
last_autosave_timestamp  
autosave_slot = "autosave_end_day"  
manual_save_slots  
  
最低实现：  
  
只需要 1 个自动保存槽即可。  
后续可以扩展为最近 3 天滚动 autosave，但 V1 不强制。

```

unassign <人数/all> from <目标>

```

```
设置目标岗位人数，不是在原有人数上追加。
```

1. 人口类型是否合法。  
2. 目标建筑是否存在。  
3. 目标是否允许该人口类型。  
4. 是否需要成人兜底。  
5. 是否处于事件临时限制。  
6. 是否超过岗位容量。  
7. 是否因天气 / 第七霜落禁止外勤。  
   
⸻  
   
## 5.4 炉心命令  
```
furnace off
furnace 1
furnace 2
furnace 3

```
规则：  
1. 玩家可在 end_day 前反复调档。  
2. 以 confirm end_day 前最终有效档位为准。  
3. furnace off 高风险，需要 confirm。  
4. 燃料不足时不模拟半日燃烧。  
5. 供暖不足按可支付最高完整档位或 furnace off 结算。  
   
⸻  
   
## 5.5 过载命令  
```
overload off
overload 1
overload 2

```
规则：  
1. furnace off 不可过载。  
2. 过载增加建筑有效温度。  
3. 过载增加煤耗。  
4. 过载增加炉心压力。  
5. 炉心压力红线后继续过载并再次 end_day，可触发炉心崩毁。  
6. 过载不能用 woodfuel 支付。  
   
⸻  
   
## 5.6 heat 命令  
```
heat <建筑名或编号>

```
规则：  
1. heat 开局可用。  
2. heat 单次耗煤 20。  
3. heat 当日生效。  
4. 日结后关闭。  
5. 同一建筑同日不可重复 heat。  
6. heat 不计入今日完整日耗。  
7. heat 不返煤。  
8. 住宅不可 heat。  
9. 研究所默认不可 heat。  
10. 应急加热装置是 heat 强化科技，不是解锁科技。  
11. woodfuel 不可支付 heat。  
   
⸻  
   
## 5.7 woodfuel 命令  
```
woodfuel
woodfuel on
burn wood
燃烧木材

```
规则：  
1. woodfuel 进入 V1。  
2. woodfuel 是应急燃料。  
3. woodfuel 只用于炉心基础供暖补足。  
4. woodfuel 转换效率远低于煤炭。  
5. woodfuel 不进入煤炭库存。  
6. woodfuel 不能替代煤炭。  
7. woodfuel 不支付 heat。  
8. woodfuel 不支付 overload。  
9. woodfuel 不改变今日完整日耗公式。  
10. woodfuel 当日手动确认，当日生效。  
11. 日结后自动关闭。  
12. 次日若仍需使用，必须再次手动确认。  
13. 煤炭不足时不得自动燃烧木材。  
确认弹窗固定：  
```
你正在确认燃烧木材作为应急燃料。

木材资源有限，燃烧效率远低于煤炭且不能替代煤炭。

是否确认？

```
PENDING：  
```
woodfuel_conversion_ratio
woodfuel_daily_limit
woodfuel_minimum_unit
woodfuel_cooldown

```
   
⸻

## 六、每日流程  
## 6.1 新一天开始  
顺序：  
```
1. current_day 已推进。
2. 检查昨日到期承诺成功 / 失败。
3. 移除已结算承诺。
4. 释放承诺槽。
5. 检查固定日期事件。
6. 检查重大事件。
7. 检查普通事件。
8. 按事件优先级与数量限制显示事件。
9. 更新 status / city / laws / tech 面板。
10. 等待玩家操作。

```
注意：  
```
承诺到期结算不排入事件优先级队列。
承诺到期结算不占重大 / 普通事件名额。

```
   
⸻

## 6.3 end_day 前预警  
必须检查：  
1. 煤炭不足。  
2. 炉心关闭。  
3. 炉心档位过低。  
4. 过载压力。  
5. woodfuel 是否开启及木材消耗风险。  
6. 食物不足。  
7. 生食食用风险。  
8. 食堂停工。  
9. 医疗站 / 医院停治。  
10. 温室停产。  
11. 住房不足。  
12. 无住所人口。  
13. 高寒暴露。  
14. 儿童高风险岗位。  
15. 长班制 / 加班日风险。

## 七、end_day 日结顺序  
推荐代码窗使用以下日结顺序。  
```
0. 锁定当日输入
1. 校验硬阻塞
2. 读取最终炉心 / 过载 / woodfuel / heat 状态
3. 支付炉心基础供暖与过载燃料
4. 判断实际炉心档位与供暖不足日
5. 结算炉心压力变化
6. 计算区域有效温度
7. 计算建筑有效温度
8. 判断建筑运行 / 停工 / 停治 / 停产 / 停业
9. 结算采集与生产
10. 结算食物链
11. 结算医疗、疾病、重症、伤残、死亡
12. 结算住房、寒冷暴露与饥饿后果
13. 结算信任 / 恐慌
14. 结算 006A / 006C 当日行动效果收尾
15. 推进研究进度并在日结末尾提交科技完成

```
```
16. 更新承诺目标状态，但不在 end_day 结算到期承诺
17. 检查硬失败
18. 检查隐藏成就
19. 记录当日日志与终局标签候选
20. 自动关闭当日效果
21. 写入 end_day 自动保存
22. 推进到下一天

```
  
自动保存规则：  
  
1. 每次成功完成 end_day 日结后，系统必须写入自动保存。  
2. 自动保存发生在当日日结、失败检查、隐藏成就检查、日志记录、当日效果关闭之后。  
3. 自动保存应保存完整 GameState、当日日志、事件历史、承诺状态、终局标签候选、随机种子 / 可复现状态。  
4. 如果 end_day 被硬阻塞，不写入新的自动保存。  
5. 如果玩家在风险确认前取消 end_day，不写入新的自动保存。  
6. 如果当日触发硬失败，应在失败结果生成后保存失败状态，便于查看终局报告。

## 7.1 燃料与供暖  
规则：  
1. 今日供暖读取 end_day 前已有煤炭。  
2. 今日生产煤炭日结后入库，主要保障明日。  
3. 今日采煤不能支付今日炉心缺口。  
4. woodfuel 可以临时补足炉心基础供暖。  
5. woodfuel 不进入煤炭库存。  
6. woodfuel 不支付 heat。  
7. woodfuel 不支付 overload。  
8. heat 成功时已即时扣煤。  
9. heat 不计入今日完整日耗。  
10. 供暖不足日不模拟半日燃烧。  
   
⸻  
   
## 7.2 温度与建筑运行  
计算顺序：  
```
区域有效温度 = 后台真实天气温度 + 炉心档位供暖修正 + 区域修正
建筑有效温度 = 区域有效温度 + 建筑保温科技修正 + 炉心功率稳定修正 + 过载修正 + heat 修正 + 终局炉心稳定修正

```
建筑阈值按 008-3 接入。  
注意：  
1. 住宅有有效温度概念，但不可 heat。  
2. 狩猎小屋普通寒冷可运行但有风险。  
3. 狩猎小屋暴雪 / 黑霜 / 第七霜落停工。  
4. 守炉堂 / 巡查所不参与温度停工。  
5. 研究所默认不可 heat。  
   
⸻  
   
## 7.3 采集与生产  
最低生产顺序：  
```
地表采集 / 资源生产
→ 狩猎小屋 / 温室产出生食
→ 食堂加工生食为熟食
→ 居民消耗食物

```
已确认：  
1. 当天狩猎产出的生食可当天加工。  
2. 当天加工出的熟食可当天食用。  
3. 食堂不能凭空生产食物。  
4. 熟食优先消耗。  
5. 熟食不足可消耗生食。  
6. 熟食与生食都不足进入饥饿系统。  
   
⸻  
   
## 7.4 医疗、疾病、死亡  
顺序：  
```
新增患病
→ 医疗容量
→ 治疗优先级
→ 恢复 / 恶化
→ 重症死亡 / 伤残
→ 冻伤伤残
→ 寒冷 / 饥饿 / 特殊死亡
→ 更新人口

```
规则：  
1. 临时救护 D1～D3 容量 5，D4 起归零。  
2. 医疗站 / 医院容量按已封存建筑口径接入。  
3. 自然死亡软上限按 008-3 接入。  
4. 承诺失败不得直接死亡。  
5. 事件不得绕过疾病 / 死亡系统。  
   6.伤残不是不会死亡。   
⸻
